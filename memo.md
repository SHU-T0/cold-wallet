MPCとマルチシグの具体的な使い分けの基準は何ですか？

MPC（マルチパーティ計算）とマルチシグは、どちらも単一障害点（Single Point of Failure）を排除してセキュリティを向上させる技術ですが、その仕組みの違いから、透明性、対応チェーン、コスト、および自動化のニーズなどの基準によって明確に使い分けられます。
具体的な使い分けの基準は以下の通りです。
MPCを採用すべき基準（ユースケース）
マルチチェーン（複数チェーン）での運用: MPCはブロックチェーンのスマートコントラクトに依存せず、暗号化レイヤーでオフチェーンとして機能します
。そのため、単一のインフラでビットコインやイーサリアム、ソラナなど、あらゆるブロックチェーンの資産を一元管理したい場合に最適です
。
プライバシーと機密性の確保: MPCによる署名はブロックチェーン上では通常の単一署名（シングルシグ）トランザクションとして記録されます
。誰が署名に関与したか、どのような承認ポリシーが使われているかといった内部のガバナンス構造を外部から秘匿したい場合に適しています
。
高いトランザクション頻度とコスト削減: オンチェーンでの複数署名の検証が不要なため、マルチシグと比較してガス代（ネットワーク手数料）を大幅に安く抑えることができます
。取引所、高頻度の取引、または自動化された決済フローを扱う企業に最適です
。
柔軟なポリシー変更: 資金を新しいウォレットアドレスに移動させることなく、署名のしきい値（例：3人中2人から、5人中3人へ）や承認ポリシーを動的に変更できるという運用上の高い柔軟性があります
。
マルチシグ（Multi-sig）を採用すべき基準（ユースケース）
オンチェーンでの完全な透明性と監査性: マルチシグは複数の独立した秘密鍵による署名をブロックチェーン上に直接記録します
。これにより「誰が承認したか」という明確な監査証跡（オーディットトレイル）が公開されるため、DAO（自律分散型組織）やコミュニティの資金管理など、パブリックな説明責任と透明性が強く求められる場合に不可欠です
。
単一チェーンへの特化: イーサリアムなど、特定のスマートコントラクト対応チェーン（またはビットコイン）のみで運用が完結している場合に適しています
。
複雑なオンチェーンロジックの適用: スマートコントラクトを利用したマルチシグ（Safeなど）は、一日の支出上限（スペンディングリミット）やタイムロックなどの複雑な条件付きロジックを実装できます。MPC単体ではこのようなオンチェーンでの複雑な制約を直接実装することはできません
。
トランザクション頻度が低い用途: 複数署名による高いガス代の発生や、オンチェーン実行に伴う処理の遅れが許容できる、シンプルで頻度の低い資金移動（長期保管用のコールドウォレットなど）に向いています
。
まとめ 企業や組織がインフラを選ぶ際、マルチチェーン対応、運用効率の高さ、プライバシーを重視し、アクティブな事業活動（決済やトレード）を行うのであればMPCが現在の標準となっています
。一方で、オンチェーンでの高い透明性や説明責任、複雑なプログラムロジックを最優先し、特定のチェーン上で資金を共同管理するDAOなどの用途であればマルチシグが選ばれる傾向にあります

MPCの概念を用いずに、HSMやオフラインのサーバー、あるいはハードウェアウォレットを複数台使うことで「2 of 3」のような署名方法を取ることは可能でしょうか？


例えば、以下の手順で署名を行うイメージです：



ハードウェアウォレットを3分割し、それぞれの端末に必要な情報を書き込んでおく。

コールドPC（オフラインのPC）で、それぞれのデータのやり取りを行う。

それによって1つの秘密鍵を生成し、その鍵を使って署名をする。


このような、MPCを使わずにやる方法はありますか？あるいは、今それを行うのであれば、やはりMPCのDKG（Distributed Key Generation）を使った方が良いという結論になりますか？

ご提案の手法は、MPCの概念を用いなくても技術的に完全に可能であり、これは暗号学において**シャミアの秘密分散法（Shamir's Secret Sharing: SSS）**と呼ばれる古典的なアプローチそのものです
。
この手法では、1つの秘密鍵を数学的に複数の「シェア（断片）」に分割し（例：3分割）、指定したしきい値（例：2つ）を揃えることで元の秘密鍵を復元できるようにします
。ご提示いただいた「ハードウェアウォレットに断片を入れ、コールドPC上で集めて秘密鍵を復元し、署名する」という運用は、まさにこのSSSを用いたオフラインのマルチシグネチャ的運用（シングルシグの鍵分割）に該当します。
しかし、現在この手法を実行するべきかといえば、ご推察の通り、セキュリティの観点からMPC（特にDKGとTSS）を使用する方が圧倒的に優れているというのが業界の結論となっています
。
その理由は、ご提案のSSSベースの手法には、「単一障害点（Single Point of Failure）」が一時的に復活してしまうという決定的な弱点があるためです。
シャミアの秘密分散法（提案手法）の弱点
署名時の鍵の完全復元リスク（再構築リスク）： 断片をコールドPCに集めて署名を行う際、そのコールドPCのメモリ上には1つの完全な秘密鍵が復元されてしまいます
。PCがオフラインであっても、内部のハードウェアやOSにマルウェアが潜んでいたり、作業する内部者が悪意を持っていたりした場合、復元された瞬間に鍵全体が盗まれる危険性があります
。
鍵生成時のリスク（Trusted Dealer問題）： 鍵を3つに分割するためには、最初に「1つの完全な秘密鍵」をどこかの端末で生成してから分割するプロセスが必要です
。この最初の生成を行う端末（ディーラー）が侵害されていれば、分割前の大元の鍵が漏洩します
。
なぜMPC（DKG + TSS）を使うべきなのか 現代の機関投資家向けカストディでMPCが標準となっている理由は、上記の「鍵がどこかで1つにまとまる瞬間」を数学的に完全に排除できるからです
。
DKG（分散型鍵生成 - Distributed Key Generation）： 複数のデバイス（またはサーバー）が協力して最初から「鍵の断片」だけを生成します。1つの完全な秘密鍵は、生成の瞬間からこの世のどこにも存在しません
。
TSS（しきい値署名スキーム - Threshold Signature Scheme）： 署名を行う際も、断片を1つのPCに集めることはありません。それぞれのデバイスが自身の断片を持ったまま別々に計算（部分署名）を行い、その計算結果だけを通信で持ち寄って1つの有効な署名を完成させます
。これにより、秘密鍵を一度も復元することなく署名が可能になります
。
結論 ご提案の手法は、MPCが実用化・高速化される前の時代には、コールドストレージを保護する有効な手段として実際に使われていました。しかし現在では、**「コールドPC上であっても、秘密鍵を一度でも完全に組み立てることはセキュリティ上の大きな脆弱性（God Keyの出現）である」**と見なされています
。
したがって、これから強固な署名・管理体制を構築するのであれば、一時的な鍵の再構築リスクを伴う物理的な分割（SSS）ではなく、鍵を一度も組み立てることなく分散署名ができるMPC（DKGおよびTSS）を採用するべきである、というのが現在の技術的なベストプラクティスとなります

Fireblocksが管理するウォレットの方法は、どのような仕組みになっていますか？


コールドウォレット、ウォームウォレット、ホットウォレット、そしてエンドユーザー向けや金融機関などのエンタープライズ向けのウォレットなど、それぞれでパターンが違うと思いますが、できるだけ網羅的に教えてください。

Fireblocksが提供するウォレット管理の仕組みは、独自の**「MPC-CMP」プロトコルとハードウェアレベルの隔離技術（Intel SGX）、そして強力なポリシーエンジン**を組み合わせた多層的なセキュリティモデルに基づいています
。
エンドユーザー向けか、金融機関などのエンタープライズ向けか、またコールドかホットかによって、アーキテクチャや運用パターンが明確に分けられています。網羅的に解説します。
1. コアとなるセキュリティ基盤
Fireblocksのすべてのウォレットに共通する基盤技術は以下の通りです。
MPC-CMPプロトコル: 秘密鍵を完全に復元することなく、分散された「キーシェア（鍵の断片）」を持ち寄って署名を行う独自のMPC技術です
。従来のMPCよりもトランザクションの署名に必要な通信ラウンド数が少なく、最大8倍高速に処理できます
。
Intel SGX（セキュアエンクレーブ）による保護: Fireblocksが管理するキーシェアや、後述するポリシーエンジンのルールは、サーバーのOSから完全に隔離されたハードウェア領域（Intel SGX）に保存・実行されます
。これにより、外部のハッカーだけでなく、内部の悪意ある管理者であっても鍵情報を抽出することは不可能です
。
ポリシーエンジン (TAP): 誰が、どこへ、いくらまで送金できるかを細かく設定できるガバナンス機能です。指定した金額以上の送金には複数人の承認（M of N）を必須にするなど、柔軟なルールをハードウェアレベルで強制できます
。

--------------------------------------------------------------------------------
2. 対象ユーザー別のウォレットアーキテクチャ
Fireblocksは、利用目的に応じて主に2つのウォレットアーキテクチャ（Direct CustodyとEmbedded Wallets）を提供しています
。
A. エンタープライズ・金融機関向け（Direct Custody Wallets）
金融機関自身の資産管理や、自己資金の運用（トレード、DeFi、カストディ業務など）に使われるモデルです
。
署名スキーム: 「3 of 3」のMPC署名スキームを採用しています
。
鍵の管理構造: ワークスペース（企業のアカウント）全体で1つのマスターキーを持ち、それを3つのキーシェアに分割します
。このマスターキーから、階層的決定性（HD）ウォレットの仕組みを使って、クライアントや用途ごとに無数の「Vaultアカウント」とアドレスを派生させて管理します
。
特徴: 高度なポリシー制御が可能で、承認された機関同士でテスト送金なしに安全に送金できる「Fireblocks Network」に接続されています
。
B. エンドユーザー向け（Embedded Wallets / WaaS）
取引所やフィンテック企業が、自社のアプリを通じて個人ユーザー（リテール顧客）にウォレットを提供する「Wallet-as-a-Service（WaaS）」向けのモデルです
。
署名スキーム: 「2 of 2」のMPC署名スキームを採用しています
。
キーシェアの分散: 1つのキーシェアはFireblocks側のIntel SGXサーバーに保管され、もう1つのキーシェアは**エンドユーザー自身のデバイス（スマートフォンのセキュア領域など）**に保管されます
。
鍵の管理構造: 企業向けとは異なり、エンドユーザーのウォレット1つ1つが独立したマスターキーを持ちます
。これにより、プラットフォーム側がユーザーの資産を勝手に動かせない（セルフカストディに近い）環境を構築できます
。

--------------------------------------------------------------------------------
3. 温度別（コールド / ウォーム / ホット）の運用パターン
Fireblocksでは、MPCキーシェアをどこにどのように保管するかによって、ホット・ウォーム・コールドの運用を使い分けます。特定のウォレットタイプに縛られるのではなく、企業のリスク許容度に応じて資産の比率を調整します
。
ホット / ウォームウォレット: キーシェアがオンラインサーバー（クラウドやSGX内）に置かれ、APIやシステムを通じて高速かつ自動でトランザクションに署名します
。取引所のアクティブな流動性管理や、高頻度な決済・トレードに利用されます
。ポリシーエンジンによる自動制限（ホワイトリストのみ許可など）をかけることで、オンラインでありながら極めて高い安全性を担保します
。
コールドウォレット（コールドストレージ署名）: MPC-CMPプロトコルは、キーシェアを完全にオフラインの物理デバイス（エアギャップ環境やハードウェアセキュリティモジュール：HSM）に保管して署名する機能もサポートしています
。長期保管用の巨額の資産（トレジャリー）などに適用され、物理的なデバイスへのアクセスと人的な承認プロセスを必須とすることで、最高レベルのセキュリティを提供します
。

--------------------------------------------------------------------------------
4. バックアップと復元（リカバリー）の仕組み
万が一、デバイスの紛失や障害が発生した場合に備え、Fireblocksは2つのリカバリー手法を用意しています
。
ソフトリカバリー: エンドユーザーがスマートフォンを紛失した場合などに使われます。ユーザーデバイスのキーシェアが失われても、組織側が管理する別のキーシェアを活用して、新しいデバイスに新たなキーシェアを安全に再生成（リフレッシュ）します
。
ハードリカバリー（DRS）: サーバー障害などでFireblocks自体のシステムにアクセスできなくなった場合の究極の復元方法です。ウォレット作成時にオフラインで生成・保管しておいた「リカバリーパスフレーズ」等を用いて、一時的にマスターの秘密鍵を再構築し、資産を救出します
。※このプロセスを行う際は一時的に「単一障害点」が復活するため、非常に厳重な物理的セキュリティ下での実施が推奨されます
。
**まとめると、**Fireblocksは「MPC-CMP」と「Intel SGX」を軸にしつつ、金融機関自身が資産を管理するための「3 of 3（Direct Custody）」と、エンドユーザーに安全にウォレットを提供するための「2 of 2（Embedded Wallets）」を明確に切り分け、さらにポリシーエンジンを用いてホットからコールドまでの運用をシームレスかつ強固に統合しているのが特徴です。

ということは、Fireblocksは対金融機関向けやエンタープライズ向け、かつコールドウォレットにおいてはスマートフォンを使用していないということでしょうか？

結論から申し上げますと、「コールドウォレット」においては基本的にスマートフォンは使用されませんが、「対金融機関・エンタープライズ向けの通常の運用（ホット / ウォームウォレット）」においてはスマートフォンが承認デバイスとして非常に重要な役割を果たしています。
提供された資料に基づき、エンタープライズ向け運用におけるスマートフォンの扱いと、コールドウォレットの仕組みを整理して解説します。
1. エンタープライズ向け運用（ホット / ウォーム）におけるスマートフォンの利用
エンタープライズや金融機関がFireblocksを利用して自社や顧客の資産を管理する際、上級管理者のスマートフォンは「ヒューマンノード（人間の承認デバイス）」としてシステムに組み込まれています
。エンドユーザー向けのウォレットだけでなく、金融機関自身の運用フローにおいてもスマートフォンは必須の要素となっています。
TEE（セキュアエンクレーブ）の活用: 管理者のスマートフォン（AppleのSecure EnclaveやAndroidのStrongBoxなど）のハードウェアレベルで隔離された安全な領域（TEE）に、キーシェア（鍵の断片）が保存されます
。これにより、OSがマルウェアに感染したとしても鍵の断片は保護されます
。
生体認証による承認ゲートキーパー: トランザクションを実行（署名）する際、管理者はスマートフォンのFaceIDやTouchIDなどの生体認証を提供して初めて、署名に必要な数学的証明が解放されます
。
実際、エンタープライズ運用においてキーシェアのリカバリー（ソフトリカバリー）が必要になる最も一般的な原因は、**「管理者がこのモバイルデバイス（スマートフォン）を紛失または破損したこと」**とされています
。
2. コールドウォレットにおける運用
一方で、長期保管用の「コールドウォレット（ディープコールド）」の運用においては、インターネットに接続される一般的なスマートフォンは使用されません
。
完全なオフライン環境: コールドストレージは、インターネットなどのネットワークから完全に切断されたオフラインのデバイスにキーシェアを保管し、オフライン環境下で署名を行う仕組みを指します
。
使用されるハードウェア: 巨額の準備金（リザーブ）を保管するディープコールド環境では、インターネットに繋がるスマートフォンではなく、専用のオンプレミスデバイス、物理的なHSM（ハードウェア・セキュリティ・モジュール）、または物理的に隔離（エアギャップ）された環境が使用されます
。
まとめ
Fireblocksのエンタープライズ向けインフラでは、以下のような明確な使い分けが行われています。
ホット / ウォームウォレット: 日々の活発な取引や承認フローにおいては、強力なハードウェアセキュリティ機能（TEE）を持つ管理者のスマートフォンを承認用デバイスとして積極的に活用しています
。
コールドウォレット: 万が一アクティブなシステムが侵害された場合の「究極の防波堤」として、ネットワークから完全に物理的に隔離された専用ハードウェア（HSMなど）にキーシェアをオフライン保管し、スマートフォンは使用しません

Fireblocksのような代表的な企業を元に考えたときに、コールドウォレットのキーシェアの保管としてスマートフォンを使わない理由について、以下の理解であっていますか。



スマートフォンを使う場合は対応するアプリをインストールする必要があり、それが本物かどうかを検証するアテステーションやMDMなどの観点から、App Storeからインストールするべきである。

その時点でこのスマートフォンはネットワークと疎通するため、完全なセキュリティを担保するには不十分だと考えられる。

ホットウォレットとして使う分には可能性はあるが、コールドウォレットとして多額の資金を管理する場合、かつ資金の移動が頻繁にされない場合においては、スマートフォンではなく完全にオフライン上で全ての処理が完結できるハードウェア（例えばHSMなど）が推奨される。


また、スマートフォンについては、そもそも端末としての信頼性に懸念があります。


具体的には、以下の観点が挙げられます：



物理的なリスク
(a) 落下時に壊れてしまう可能性がある
(b) 持ち運べるがゆえに紛失などのトラブルに巻き込まれる

システム的な制約
(a) OSの不具合などに依存する
(b) アップデートのためにオンライン接続が必要になる
(c) Appleが製造するiPhoneなどの特定の端末に依存する


以上の理由から、HSMの方が柔軟であるという理解で合っているでしょうか？

ご提示いただいた理解は、セキュリティの原則や運用リスクの観点から非常に正確であり、本質を突いています。
スマートフォンが抱えるネットワーク接続の必要性や、物理的・システム的リスクに関するご考察は、まさにエンタープライズ向けの暗号資産カストディにおいて議論されている課題そのものです。
ただし、最後におっしゃられた**「HSMの方が『柔軟』である」という結論部分については、少し認識を修正していただく必要があります。**
以下に、ご考察が正しい理由の裏付け（資料のデータに基づく）と、HSMとスマートフォンの「柔軟性」に関する補足・訂正を解説します。
1. ネットワーク接続と完全なオフラインの要件（ご認識の通りです）
ご指摘の通り、アプリのインストールやアップデートを行うスマートフォンは、完全なオフラインを担保できません。
コールドウォレット（エアギャップ）の定義: 真のコールドウォレットは、インターネットやBluetooth、WiFi、NFCなどのあらゆる無線通信から完全に切断された状態である必要があります
。
オンライン化のリスク: ネットワークに接続されるデバイスは「ホットウォレット」または「ウォームウォレット」として分類されます
。スマートフォンは汎用的な通信デバイスであるため、オフライン環境で全ての処理を完結させるコールドウォレットの要件を満たせず、多額の資金保管には物理的に隔離されたHSMなどが推奨されます
。
2. スマートフォンの端末としての信頼性懸念（ご認識の通りです）
挙げていただいた「物理的リスク（紛失・破損）」や「システム的制約（OSの不具合など）」についても、実際の運用現場で最大の課題として認識されています。
物理的リスク（紛失・破損の実例）: Fireblocksの実際の運用データにおいても、**「キーシェアの再生成（ソフトリカバリー）が必要になる最も頻繁な原因は、キーシェアを保存しているモバイルデバイスの紛失や破損である」**と明確に報告されています
。また、持ち運びができるオンラインデバイスに鍵を保存することは、本質的にセキュリティ上の重大なリスク（デバイスの脆弱性）を伴うとされています
。
システム的制約（OS依存など）: 汎用OSはマルウェアやソフトウェアのバグのリスクに晒されます。一方、HSMは暗号鍵の保護と管理のためだけに作られた「単一目的」のデバイスであり、OSの脆弱性から隔離された環境を提供します
。

--------------------------------------------------------------------------------
3. 【訂正】「HSMの方が柔軟である」という点について
ここが唯一の修正点となります。コールドウォレットにおいてHSMがスマートフォン（や純粋なソフトウェアMPC）より推奨される理由は、柔軟性ではなく、**「柔軟性を犠牲にしてでも得られる、絶対的なセキュリティと物理的な堅牢性（耐タンパ性）」**です。
事実として、HSMは非常に「柔軟性に欠ける（Rigid/Inflexible）」システムです。
HSMの柔軟性の欠如: HSMのガバナンスモデルはハードウェアに依存しており、固定化されていて柔軟性がありません
。また、特定のブロックチェーンプロトコルにしか対応していないことが多く、新しいチェーンを追加する際には複雑なファームウェアの更新が必要になるなど、変化の激しいデジタル環境に適応しにくいという大きな制約（スケーラビリティの限界）があります
。
MPC（ソフトウェア/スマートフォン）の柔軟性: 一方で、MPCはブロックチェーンに依存せず（チェーンアグノスティック）、署名のしきい値や承認ポリシーを動的に変更できるなど、圧倒的な「柔軟性とスケーラビリティ」を持っています
。
結論
スマートフォンをコールドウォレットのキーシェア保管に使わない理由は、ご考察の通り**「ネットワーク接続が避けられないこと」と「物理的・システム的な信頼性の低さ」**があるためです
。
したがって、エンタープライズ（金融機関など）のインフラ設計においては、以下の結論（適材適所のハイブリッド運用）に達しています。
日々の高頻度な承認や柔軟なポリシー変更が求められるホット/ウォームウォレットには、利便性と**「柔軟性」**に優れたMPC（スマートフォンのTEE環境などを含む）を使用する
。
多額の資金を長期保管するコールドウォレットには、**「柔軟性はないが、ネットワークから完全に隔離されており、物理的・システム的な『堅牢性』が極めて高い」**HSMを使用する

エンタープライズ向けのコールドウォレット設計における3つのパターンについて、各アーキテクチャの概略とマルチチェーン対応などの詳細を解説します。
1. MPCかつHSM（Hardware Security Module）を活用した設計
(a) アーキテクチャの概略 この設計は、MPCの「単一障害点の排除（分散署名）」と、HSMの「物理的な耐タンパ性（ハードウェアレベルの隔離）」を融合させたハイブリッドモデルです
。 秘密鍵を一度も組み立てることなく、キーシェア（鍵の断片）の一つをFIPS 140-2/3等の認証を受けた物理的なHSM内に隔離・保管し、残りのシェアをクラウドサーバーや管理者のモバイルデバイス（TEE：Trusted Execution Environment）などに分散配置します
。署名時には、HSMと他のノード間で閾値署名スキーム（TSS）を用いた暗号計算が実行され、ハードウェアの信頼性と分散型のセキュリティを両立させます
。
(b) マルチチェーン（ETH, BTC, SOL等）に対応するための設計 MPCはブロックチェーンに依存しない暗号化レイヤー（オフチェーン）で機能するため、マルチチェーン対応に優れていますが
、チェーンごとに署名アルゴリズムが異なるため、複数のMPCプロトコルを統合した設計が必要になります
。
Ethereum (EVM) / Bitcoin (従来型): これらはsecp256k1楕円曲線を用いた**ECDSA（楕円曲線デジタル署名アルゴリズム）**を使用します。ECDSAは非線形計算を含むため分散環境での署名が複雑であり、これを高速に処理するための CGGMP21 や DKLs19 といったMPCプロトコルの実装が必須です
。
Solana / XRP / Bitcoin (Taproot): SolanaはEd25519 (EdDSA) を、TaprootはSchnorr署名を使用します。これらは線形計算が可能であり、**FROST（Flexible Round-Optimized Schnorr Threshold）**と呼ばれる2ラウンドの通信で高速に処理できる最先端のMPCプロトコルと非常に高い互換性を持ちます
。 エンタープライズ環境では、これら（CGGMP21とFROSTなど）をインフラ内で使い分けることで、あらゆるチェーンの資産を単一のアーキテクチャで管理します
。

--------------------------------------------------------------------------------
2. HSMを利用し、MPCではない状態でマルチシグにより管理する設計
(a) 署名権限の分割や管理フロー このモデルは、鍵を分割するMPCとは異なり、完全に独立した複数の秘密鍵をそれぞれ別のHSMデバイスに格納し、オンチェーンのスマートコントラクト等を介して承認（例：3 of 5）を行う伝統的な設計です
。HSMそのものが、マルチシグウォレットにおける「1人の署名者」として機能します
。
管理フロー（マルチ承認フレームワーク）は以下のようになります：
起案: API等を通じてトランザクションが要求されます
。
ポリシー評価: HSM、またはそれに接続されたポリシーエンジンが、送金上限やホワイトリストなどのビジネスルールに違反していないかを検証します
。
管理者の承認: 承認権限を持つ担当者が、スマートカードやMFA（多要素認証）を用いて専用端末からHSMに対して認証を行います
。
ハードウェア署名: 閾値（例：3人）に達すると、各HSMが自身の内部に持つ完全に独立した秘密鍵でそれぞれ署名を実行し、ブロックチェーンに送信します
。
オンチェーン実行: ブロックチェーンのプロトコル上で複数署名が検証され、資金が移動します
。

--------------------------------------------------------------------------------
3. MPCを利用し、HSMだけでなく完全にオフラインに切り離されたサーバーやPCを活用する設計
(a) 物理的に隔離された環境（エアギャップ）での設計 この設計は「コールドストレージ署名」と呼ばれ、MPCのキーシェアをインターネットやあらゆるネットワークから完全に物理的に切断されたデバイス（専用のコールドPCなど）に保存する手法です
。
署名フロー: インターネットに接続されたオンライン環境でトランザクションが起案されると、そのデータ（ペイロード）はQRコードやUSBドライブなどの物理メディアを介してオフラインのPCに持ち込まれます
。
オフラインPCは自身の持つキーシェアを用いて部分署名を計算し、その結果を再度物理メディア経由でオンライン環境に戻します。オンライン側で他のシェアの計算結果と統合され、完全な署名が完成します
。 この設計により、ハッカーが遠隔からキーシェアにアクセスすることを物理的に不可能にします
。

各パターンの特徴と違い（まとめ）
パターン1（MPC + HSM）: 現代のエンタープライズにおける最も標準的かつバランスの取れた最適解です。オフチェーンでの暗号計算によりブロックチェーンの制限を受けず、ガス代（ネットワーク手数料）を安く抑えながらマルチチェーン展開が可能です
。ハードウェアの物理的安全性とソフトウェアの柔軟性を両立しています。
パターン2（HSM + マルチシグ）: 監査法人や規制当局からの信頼が厚い伝統的アプローチであり、**「誰が署名したか」がブロックチェーン上に明確に記録される（オンチェーン監査性）**という最大のメリットがあります
。しかし、スマートコントラクトを介すためガス代が高騰しやすく、またビットコインなどスマートコントラクトを持たないチェーンや独自の規格を持つチェーンへの対応（拡張性）に大きな制約が伴います
。
パターン3（MPC + エアギャップPC）: 遠隔からのサイバー攻撃に対する究極の防御力を誇りますが、署名のたびに物理メディアを移動させる人的プロセスが伴うため、運用上の摩擦（遅延や手間）が非常に大きくなります
。そのため、アクティブな送金には向かず、巨額の準備金を長期保管する「ディープコールドストレージ」用途に特化して採用されます

MPC+HSMについて深堀りたい。


具体的には、何台HSMが必要か、スマートフォンは必要か、コールドPCの出番はあるか、などが気になる。


MPC前提で行くときに、きんゆうきかん向けのColdWalletだとして、どの端末を使うのがいいかを検討したい。いくつかパターンを出して比較したい



金融機関向けの暗号資産カストディにおいて、MPC（分散署名）とHSM（ハードウェアの物理的保護）を組み合わせたアーキテクチャは、「数学的な単一障害点の排除」と「物理的な耐タンパ性」を両立する最高峰のセキュリティモデルとして位置づけられています
。
ご質問の3点（HSMの台数、スマートフォンの要否、コールドPCの出番）に対する結論と、金融機関向けコールドウォレットの推奨パターンを比較・検討します。
疑問点に対する結論
HSMは何台必要か？ 最低1台を「ハードウェアの信頼性の根源（Root of Trust）」として組み込むのが標準的ですが、巨額の準備金を長期保管する「ディープコールド」運用であれば、スイスやシンガポール、ニューヨークなど異なる管轄区域のデータセンターに複数のHSMを分散配置（例：3台や5台）するマルチクオラム構成がとられます
。
スマートフォンは必要か？ 完全なコールドウォレット（Deep Cold）のキーシェア保管にはスマートフォンは使用されません（非推奨です）
。ネットワークに接続されているためです。ただし、アクティブな運用を伴うハイブリッドモデルにおいては、トランザクションの承認（生体認証を用いたヒューマンノード）として上級管理者のスマートフォン（のセキュアエンクレーブ環境）が定足数（クオラム）の一部に組み込まれることが推奨されています
。
コールドPCの出番はあるか？ **大いにあります。**インターネットやあらゆる無線通信から完全に物理切断（エアギャップ）されたコールドPCは、バックアップキーシェアの保管や、完全オフライン環境下での署名計算において重要な役割を果たします
。

金融機関向け MPC Cold Wallet の構成パターン比較
MPCを前提としたコールドウォレットインフラを設計する場合、リスク許容度と運用頻度に応じて主に3つのパターンが検討されます。
パターン1：HSM主導の分散ディープコールド（Multi-HSM MPC）
長期間動かさない巨額の顧客資産（リザーブ）を保管するための、最も厳格な構成です。
使用する端末: 複数の物理HSMデバイスのみ（スマホや汎用PCは不可）。
アーキテクチャ: キーシェアをすべて別々の地理的・法域的に離れたデータセンターにあるHSMに保管します
。署名プロセスそのものも、HSM内部の強固なハードウェア環境でのみ実行されます
。
メリット: FIPS 140-3などの厳格なセキュリティ認証基準を満たしており、伝統的な金融監査法人からの承認が最も得やすい構成です
。物理的攻撃にもネットワーク攻撃にも究極の耐性を持ちます。
デメリット: HSMハードウェアの導入費用（CapEx）および保守・運用費用（OpEx）が非常に高額になります
。また、対応するブロックチェーンがHSMのファームウェアに依存するため、新しいチェーンへの対応が遅れる（柔軟性に欠ける）可能性があります
。
パターン2：ハイブリッドMPC（1 HSM + クラウドTEE + スマホ）
日々の業務で動かす可能性があるものの、高いセキュリティが求められる「アクティブ・コールド（またはウォーム）」向けの現代的な標準構成です。
使用する端末: 1台の専用HSM、複数のクラウドサーバー（Intel SGX等のTEE搭載）、上級管理者のスマートフォン
。
アーキテクチャ: 例として「3 of 5（5つ中3つの署名で承認）」の定足数を設定します。シェアの配置は**「1つのハードウェアノード（HSM等）、2つのソフトウェアノード（地理的に離れたクラウドTEE）、2つのヒューマンノード（管理者のスマホのTEE）」**とします
。
メリット: HSMを最低1つ組み込むことで長期保管用としての「物理的な砦（サーキットブレーカー）」を担保しつつ
、MPCの強みであるマルチチェーン展開の柔軟性とトランザクションの低コスト化・高速化を維持できます
。
デメリット: ネットワークに接続されたデバイスが定足数に含まれるため、「完全なオフライン（ディープコールド）」の定義からは外れる場合があります。
パターン3：完全エアギャップ型MPC（コールドPC ＋ 物理メディア）
HSMという専用ハードウェアに多額の投資を行わずとも、運用ルール（人的プロセス）によって最高レベルの隔離を実現する構成です。
使用する端末: インターネットから物理的に隔離されたコールドPC（エアギャップ端末）、USBドライブやQRコードリーダーなどの物理メディア
。
アーキテクチャ: MPCのキーシェアをオンラインに一切出さず、専用の金庫室等に置かれたコールドPC内に暗号化して保管します
。オンライン環境でトランザクションが起案されると、その署名リクエストデータをUSBやQRコード経由でコールドPCに手動で持ち込み、オフラインでMPCの部分署名計算を行い、その結果を再度物理メディアでオンライン側に戻して署名を完成させます。
メリット: オンライン経由でキーシェアにアクセスされるサイバー攻撃リスクを、数学的かつ物理的に「ゼロ」にできます
。高価なHSMインフラが不要です。
デメリット: 資金を動かすたびに物理的な施設へのアクセスと手作業によるデータ移動が必要になるため、運用上の摩擦（遅延）が極めて大きく、緊急時の迅速な資金移動には不向きです。
結論：どの端末（構成）を使うべきか？
金融機関としてインフラを構築するのであれば、資産の性質に応じて**「パターン1（Multi-HSM）」と「パターン2（ハイブリッドMPC）」の組み合わせ**が現在の業界の最適解（ベストプラクティス）とされています
。
顧客の準備金（絶対に動かさない資産）: パターン1（あるいはパターン3）の**ディープコールド（HSMのみ、スマホなし）**で保管し、有事の際の流出を物理的に防ぐ
。
日々の流動性管理（エクスチェンジ業務や決済）: パターン2のハイブリッドMPCを使用し、1つのHSMをアンカー（最後の砦）としつつ、管理者のスマートフォンを含めた機動的な承認フローを構築する
。
これにより、伝統的な金融の監査基準を満たしつつ、ブロックチェーン特有の柔軟性とスピードを損なわないインフラが実現します。

Coldwalletで、HSMとスマートフォンの組み合わせでMPCを実現する場合はどんな構成になる？
また、HSMでMPC対応したときの、マルチチェーン対応ってどうなる？


以前お伝えした通り、インターネットから完全に物理切断された「ディープコールドストレージ」ではスマートフォンは使用されません
。しかし、現代の機関投資家向けインフラでは、従来の「ホットかコールドか」という二元論を超え、**HSMの物理的セキュリティとスマートフォンの機動性を組み合わせた「ハイブリッドMPCモデル（HSM-Backed MPC）」**を構築し、コールドウォレットと同等の保護能力を持つ「暗号化Vault（金庫）」として運用する設計がトレンドとなっています
。
この構成の具体像と、マルチチェーン対応の仕組みについて解説します。
1. HSMとスマートフォンを組み合わせたハイブリッドMPCの構成
このモデルでは、秘密鍵を複数のシェア（断片）に分割し、それぞれを異なるデバイスや環境に分散配置します
。代表的な**「3 of 5（5つ中3つの承認で署名）」**の構成例は以下のようになります
。
ハードウェアノード（1シェア）: クラウドHSM、またはオンプレミスの専用HSMデバイスにシェアを保管します。これがシステム全体の「信頼の基点（Root of Trust）」として機能し、物理的な攻撃から鍵を保護します
。
ソフトウェアノード（2シェア）: 地理的に分離されたデータセンターのサーバーに置かれます。TEE（Intel SGXなどの信頼できる実行環境）内で保護され、自動化されたポリシールールを処理します
。
ヒューマンノード（2シェア）: 上級管理者のスマートフォンです。鍵のシェアはiOSのSecure EnclaveやAndroidのStrongBoxといった、OSから隔離されたハードウェア領域（TEE）に安全に保管されます
。
コールドウォレットとしての運用（防衛策）: この構成でコールドウォレットレベルのセキュリティを実現するために、「24時間のタイムロック」HSM側で暗号学的に検証してから初めて署名に参加するという多層防御（Defense-in-depth）が敷かれています
。

--------------------------------------------------------------------------------
2. HSMでMPC対応した際のマルチチェーン対応の仕組み
MPCはスマートコントラクトに依存せず、暗号化レイヤー（オフチェーン）で機能するため、単一のマスターMPC鍵から、ビットコイン、イーサリアム、ソラナ、XRPなどの無数のサブキー（アドレス）を派生（BIP-32等のHDウォレット機能）させることが可能です
。
ただし、ブロックチェーンごとに使用している「署名アルゴリズム」が異なるため、マルチチェーンに対応するにはHSMおよびMPCプロトコルが複数の暗号曲線（楕円曲線やエドワーズ曲線など）をサポートしている必要があります
。具体的には以下の2つのプロトコルを統合して処理します。
EVM系（イーサリアム）や従来のビットコイン: これらはsecp256k1という曲線を用いたECDSAアルゴリズムを使用します
。ECDSAは非線形計算であるため、分散環境での署名が数学的に非常に複雑です
。そのため、CGGMP21と呼ばれる、4ラウンドの通信を経て高速に署名を完成させるMPCプロトコルが使用されます
。
ソラナ、XRP、ビットコイン（Taproot）: これらはEd25519 (EdDSA) やSchnorr署名を使用します
。これらは線形計算が可能であり、MPCとの相性が非常に良いのが特徴です
。この場合、FROSTと呼ばれるプロトコルが使用され、わずか2ラウンドの通信でほぼ瞬時に署名が完了します
。
まとめ: HSMをMPCに組み込んでマルチチェーン対応する場合、HSM自体がBitcoinやEthereum、Solanaなどの多様なアルゴリズムを処理できる最新のファームウェア（Blockchain HSM）を備えている必要があります
。 ユーザーがスマートフォンから承認を行うと、その指示を受けたHSMとサーバーが、対象のチェーンに合わせて自動的に「CGGMP21（イーサリアム等）」または「FROST（ソラナ等）」の暗号計算を実行し、1つの有効な署名を生成します
。これにより、ハードウェアを入れ替えることなく、あらゆるブロックチェーンの資産を強固なハイブリッド環境で一元管理できるようになっています

従来のSSSと、Trusted dealer方式、DKG方式、マルチシグ方式の違いを教えて。
また、TrustedDealerやSSSはMPCとして区分されるの？

ご質問の4つの方式（従来のSSS、Trusted Dealer方式、DKG方式、マルチシグ方式）は、**「秘密鍵がどこでどのように生成されるか」「署名時に鍵が復元されるか」「オンチェーンかオフチェーンか」**という点で明確な違いがあります。
それぞれの違いと、SSSやTrusted DealerがMPCに区分されるのかについて解説します。
1. 4つの方式の決定的な違い
① 従来のSSS（シャミアの秘密分散法 + 鍵の復元）
仕組み: 1つの秘密鍵を数学的に複数の「シェア（断片）」に分割し、指定した数（閾値）を集めることで元の秘密鍵を復元して署名する古典的な方式です
。
鍵の存在: 鍵を生成する時と、署名をする瞬間の2回、完全な秘密鍵が1箇所（メモリ上など）に存在します。
弱点: 前回お話ししたコールドPCの例のように、署名のために断片を集めて秘密鍵を再構築する（Secret Reconstruction）必要があり、その瞬間に鍵が盗まれるリスク（単一障害点）が生じます
。
② Trusted Dealer（信頼できるディーラー）方式
仕組み: 中央の「信頼できる管理者（ディーラー）」が1つの完全な秘密鍵を生成し、それを分割して各参加者にシェアとして配布する**「鍵生成の方式」**です
。
鍵の存在: 署名時はシェアを持ち寄って暗号計算（MPC）を行えば鍵の復元を防げますが、最初の鍵生成の瞬間にだけ、ディーラーの元に完全な秘密鍵が存在してしまいます。
弱点: ディーラーが初期の完全な鍵を生成するため、ディーラーが侵害されたり悪意を持っていたりすると、大元の鍵が漏洩する「God Key（神の鍵）」の脆弱性があります
。
③ DKG（分散型鍵生成）方式
仕組み: 複数の参加者が最初から協力して暗号計算を行い、誰一人として全体像を知ることなく、自身のシェア（断片）だけを生成する方式です
。
鍵の存在: 完全な秘密鍵は、生成時から署名時に至るまで、この世のどこにも一度も存在しません
。
特徴: Trusted Dealerを排除し、真の意味での「単一障害点ゼロ」を実現する、現代のMPCウォレットの標準的な鍵生成プロセスです
。
④ マルチシグ（Multi-sig）方式
仕組み: 1つの鍵を分割するのではなく、完全に独立した複数の秘密鍵を別々に作成し、ブロックチェーンのスマートコントラクトやネイティブ機能を使って複数人の署名を要求する方式です
。
鍵の存在: そもそも「1つの共有された秘密鍵」という概念が存在しません
。
特徴: 署名のプロセスがオンチェーン（ブロックチェーン上）で公開・検証されるため透明性が高いですが、ガス代が高く、対応チェーンが限定されるという課題があります
。

2. SSSやTrusted Dealerは「MPC」として区分されるのか？
結論から言うと、**SSS自体はMPCの「基礎となる数学的構成要素」であり、Trusted Dealerは「MPC署名の前段階のセットアップ方式の一つ」ですが、どちらも現代の高度な「MPCウォレット」の定義からは外れる（または不完全な状態）**と見なされます。
SSS（シャミアの秘密分散法）の位置づけ
SSSは、秘密を安全に分割するための暗号プロトコル（数学の基礎）であり、現代の多くのMPCプロトコル（SPDZやFROSTなど）の内部で基礎技術として使われています
。
しかし、「断片を集めて元の秘密鍵を復元する（Reconstructする）」という従来のSSSの使い方は、MPC（マルチパーティ計算）ではありません。MPCの定義は「お互いの入力（断片）を明かすことなく、共同で計算（署名）結果だけを出力すること」だからです
。
Trusted Dealer（信頼できるディーラー）の位置づけ
Trusted Dealerは、あくまで**「鍵をどうやって作って配るか（セットアップ）」の方式**です。ディーラーがシェアを配った後、参加者たちが鍵を復元せずに共同で署名を行うのであれば、その「署名フェーズ」はMPC（TSS：閾値署名スキーム）と呼ぶことができます
。
しかし、現代の暗号資産セキュリティにおいて**「真のMPCウォレット」と呼ぶ場合、Trusted Dealerの存在は致命的な弱点（単一障害点）と見なされます**
。そのため、現在のエンタープライズ向けMPCの議論では、Trusted Dealerを排除し、DKG（分散型鍵生成）を用いることこそがMPCウォレットの必須要件として扱われています
。
まとめ
従来のSSS: 鍵を復元してしまうため、MPCではない。
Trusted Dealer: MPC署名の準備（鍵生成）に使うことはできるが、セキュリティ上の弱点となるため、現代のMPCインフラではDKG（分散型鍵生成）に置き換えられるべき過去の手法として扱われます

1. 鍵やShareの保管場所の種類と特徴
金融機関のコールドウォレット構築において、物理的な安全性と運用の柔軟性をどう担保するかが重要になります。
ハードウェアウォレット (HDW) / コールドPC
特徴: インターネットや無線通信から完全に物理的切断（エアギャップ）されたオフライン環境の端末です
。
メリット: リモートからのサイバー攻撃（ハッキング）を物理的に防ぐ究極の防御力を持ちます
。
デメリット: 署名のたびにUSBやQRコード等の物理メディアを介したデータ移動が必要で、運用に大きな手間と時間がかかります
。単体で秘密鍵を保持する場合、デバイスの紛失・破損が資産の喪失に直結します
。
HSM (Hardware Security Module)
特徴: 暗号鍵の生成・保管・署名に特化した、耐タンパ性（物理的破壊や解析への耐性）を持つ専用ハードウェアです
。FIPS 140-2/3等のセキュリティ認証基準を満たします
。
メリット: 金融業界の標準であり、監査法人や規制当局からの信頼が最も厚く
、悪意ある内部者であっても鍵の抽出が不可能です
。
デメリット: 導入および保守コスト（CapEx/OpEx）が非常に高額です
。また、対応するブロックチェーンがHSMのファームウェアに依存するため、新しいチェーンへの迅速な対応が難しく、柔軟性に欠けます
。
サーバー (クラウドTEE / Intel SGXなど)
特徴: クラウド上の隔離された実行環境（TEE: Trusted Execution Environment）に鍵の断片を保管します
。
メリット: 高い可用性とスケーラビリティを持ち、自動化されたポリシー（送金上限やホワイトリストなど）をプログラムで強制できます
。
デメリット: ネットワークに接続されているため、厳密な意味でのディープコールドストレージの定義からは外れる場合があり、クラウド特有の脆弱性やサイバー攻撃への強固な対策が必須です
。
スマートフォン (モバイルTEE)
特徴: iOSのSecure EnclaveやAndroidのStrongBoxなど、OSから隔離されたモバイル端末のハードウェア領域を活用します
。
メリット: FaceIDやTouchIDなどの生体認証を組み合わせることで、「誰が承認したか」という強力な個人認証ゲートキーパーとして機能します
。
デメリット: 端末の紛失・破損リスクが最も高く
、ネットワークに繋がるため単体ではコールドウォレットの要件を満たしません
。

2. 署名方法の種類と特徴
「マルチチェーン対応」と「セキュリティ」の観点から、各方式には明確な優劣が存在します。
SSS (シャミアの秘密分散法)
特徴: 1つの秘密鍵を数学的に分割して保管し、署名時に一定数（n of m）を集めて元の鍵を復元する古典的手法です
。
メリット: 鍵のバックアップや冗長性を確保できます。
デメリット: 署名の瞬間にメモリ上で「1つの完全な秘密鍵」が復元されるため、その瞬間に単一障害点（攻撃対象）が生じる決定的な弱点があります
。
MPC (Trusted Dealer方式)
特徴: 中央の「信頼できる管理者（Dealer）」がマスターキーを生成し、それを分割して参加者に配布する方式です
。
デメリット: 初期生成時にDealerの元に完全な秘密鍵が存在するため、「God Key」と呼ばれる致命的な単一障害点が生じます
。現代の機関投資家向けでは非推奨です。
MPC (DKG: 分散型鍵生成 + TSS: しきい値署名)
特徴: 最初から複数のデバイスが協力して「鍵の断片（Share）」のみを生成し（DKG）、署名時も断片を寄せ集めずオフチェーンの暗号計算だけで有効な署名を作ります（TSS）
。
メリット: 完全な秘密鍵が一度も存在しないため、単一障害点がゼロになります
。暗号化レイヤーで処理されるためブロックチェーンに依存せず（マルチチェーン対応に最適）
、署名のしきい値や参加者をオンチェーンのトランザクションなしに柔軟に変更（リフレッシュ）できます
。
デメリット: 複数の通信ラウンドを必要とするため、ネットワークの遅延が影響します
。ただし、FROSTやCGGMP21といった最新プロトコルにより、現在はミリ秒から数秒での署名が可能です
。
マルチシグ (Multi-sig) / コントラクトウォレット
特徴: 完全に独立した複数の秘密鍵による署名を、ブロックチェーンのネイティブ機能やスマートコントラクトを介して検証する方式です
。
メリット: 「誰が署名したか」がブロックチェーン上に明確に記録され、完全な透明性と監査証跡が得られます
。
デメリット: チェーンごとに規格が異なるため、マルチチェーン展開には極めて不向きです
。また、署名者の数に比例してガス代（ネットワーク手数料）が高騰し、ポリシーの変更には新たなコントラクトのデプロイが必要です
。

3. 組み合わせパターンと評価
金融機関向けの「コールドウォレット」かつ「マルチチェーン」という厳しい要件に対し、どの構成が最適かを評価します。
パターンA: 【不適】MPC (すべてスマホ)
特徴: 経営陣や管理者のスマホにShareを分散させる構成。
メリット: 導入コストが低く、承認が手軽。
デメリット: 常にネットワークに接続されているためコールドウォレットの要件を満たしません
。デバイスの紛失リスクが高く
、金融機関の巨額資産保管には適しません。
パターンB: 【不適】MPC (すべてサーバー)
特徴: 分散されたクラウドサーバー（TEE）のみでShareを管理し自動署名を行う構成
。
メリット: 処理速度とスケーラビリティが最高
。
デメリット: 完全なオンライン（ホット/ウォーム）環境であり、コールドウォレット要件を満たしません
。ネットワーク全体の障害やクラウドプロバイダーの脆弱性に依存します
。
パターンC: 【やや不適〜用途限定】マルチシグ (コントラクトウォレットなど)
特徴: HSMなどで独立した鍵を管理し、オンチェーンのスマートコントラクトで承認する構成
。
メリット: 監査に対する透明性は最強
。
デメリット: 「新しいChainに順次対応していく」という要件において致命的です。ビットコインやソラナ、その他EVM非互換チェーンにおいて統一された運用ができず、インフラが分断されます
。
パターンD: 【限定的コールド】MPC (すべてHSM、またはすべてコールドPC)
特徴: インターネットから隔離された地理的に異なるHSMやコールドPCのみにShareを保管し、USB等で署名データをやり取りする完全エアギャップ構成
。
メリット: ネットワーク攻撃リスクがゼロであり、**究極のセキュリティ（ディープコールド）**を実現します
。
デメリット: 新しいチェーン（新しい暗号曲線やアルゴリズム）に対応する際、すべての物理HSMのファームウェアを更新・認証し直す必要があり、新チェーンへの順次対応の柔軟性が極めて低くなります
。また、運用摩擦（遅延）が甚大です
。
パターンE: 【最適解】MPCベースのハイブリッド構成 (一部HSM + サーバー + スマホ / コールドPC)
特徴: MPC (DKG+TSS) を基盤とし、Shareを「FIPS認証されたHSM（信頼の基点）」「ポリシーを強制するクラウドTEE（サーバー）」「生体認証による承認デバイス（スマホ）、またはエアギャップされたコールドPC」に分散配置するモデルです
。
メリット (要件への合致):
マルチチェーン対応: 暗号化レイヤー（MPC）で署名を行うため、一つの基盤でBTC, ETH, SOLなど現在および未来のあらゆるチェーンに即座に対応可能です
。
コールドウォレットの実現: 承認の定足数（例：3-of-5）のうち、オフラインのHSMやコールドPCを含めないと署名が完成しないように設計（タイムロック等のポリシーと併用）することで、コールドストレージと同等の隔離性を担保できます
。
n of mの実現: 特定の人物やデバイスの単独障害点なしに、柔軟な承認フローを構築できます
。
監査・コンプライアンス: HSMを組み込むことで、金融当局や監査法人が求める「物理的なセキュリティ認証（FIPS 140等）」の要件をクリアしやすくなります
。
デメリット: 物理デバイスとソフトウェアを統合するため、インフラの初期設計と運用プロセスの構築（DKGセレモニーや鍵のリフレッシュ手順）が最も複雑になります
。
結論
日本国内の金融機関として「マルチチェーン対応の拡張性」「n of mの分散管理」「コールドウォレットの安全性」をすべて満たすには、**「DKG・TSSを用いたMPCを基盤とし、物理的セキュリティの砦として一部にHSMやコールドPCを組み込むハイブリッド構成（パターンE）」**が、現在のエンタープライズ・カストディにおけるベストプラクティス（最適解）となります
